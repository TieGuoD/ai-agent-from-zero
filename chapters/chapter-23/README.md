## 23.1 从单 Agent 到 Multi-Agent：一段进化的故事

### 23.1.1 单 Agent 的黄金时代

让我们把时间拨回到 Agent 刚刚兴起的那段日子。那时候，我们习惯于让一个 Agent 完成所有的任务——从读取文件、分析数据到生成报告，一个 Agent 包打天下。这种"全能选手"的模式在简单的任务场景下运行得很好，就像一个经验丰富的全栈工程师，前后端通吃，数据库也搞得定。

但问题很快出现了。

想象一下，你正在开发一个智能客服系统。用户问了一个关于退货政策的问题，Agent 需要先理解用户的意图，然后去查数据库中的退货记录，接着对照公司的退货政策文档，最后生成一个既准确又友好的回复。在这个过程中，Agent 要同时扮演"理解者"、"查询者"、"政策专家"和"文案撰写者"四个角色。

一开始，你可能觉得没什么问题。但随着系统的演进，你发现越来越多的挑战涌了上来。用户开始问"我的退货和换货哪个更划算"，这需要 Agent 同时处理退货政策和换货政策，还要做财务对比分析。接着用户问"能不能帮我加急处理"，这又涉及到物流系统的对接。再后来，用户问"上次退货的时候你们承诺过免费上门取件，这次怎么不行了"，这需要 Agent 还能查询历史对话记录。

每个新功能都让这个"全能 Agent"变得越来越臃肿。它的系统提示词从最初的几百个 token 膨胀到了几千个 token，工具定义从几个变成了几十个。更糟糕的是，它的表现开始不稳定——有时候它会在理解用户意图的环节出错，影响了后续所有步骤；有时候它查对了数据但引用了错误的政策条款；有时候它生成了完美的回复但忘记了去查物流系统。

你开始意识到，一个 Agent 要同时精通所有这些领域，就像让一个人同时当律师、会计师、程序员和文案——理论上可能，但实践中极其困难。

### 23.1.2 分工的自然法则

其实，人类社会早就给出了解决方案——专业化分工。

回想一下工厂流水线出现之前，制造一辆汽车需要什么？一个全能的工匠需要自己冶炼钢铁、锻造零件、组装发动机、缝制座椅、喷涂车身。而福特的流水线把它拆分成了几十个工位，每个工位上的工人只需要精通自己的那一步骤。结果是什么？生产效率提升了数十倍，成本大幅下降。

Multi-Agent 系统的核心思想与此如出一辙：与其让一个 Agent 做所有事情，不如让多个专门的 Agent 各司其职。一个 Agent 专门负责理解用户意图，一个专门负责查询数据，一个专门负责政策解读，一个专门负责生成回复。每个 Agent 只需要精通自己的领域，系统整体的表现反而会更好。

这不是一种退步——不是因为"一个 Agent 做不好所以用多个来凑"——而是一种进化。就像从单核 CPU 到多核 CPU 的演进一样，Multi-Agent 系统代表的是更高层次的架构设计。

### 23.1.3 Multi-Agent 系统的定义

那么，到底什么是 Multi-Agent 系统（Multi-Agent System，简称 MAS）？

简单来说，Multi-Agent 系统是由多个自主的 Agent 组成的系统，这些 Agent 能够通过通信和协作来解决单个 Agent 难以有效处理的复杂问题。

这里有三个关键词需要理解：

**多个 Agent**：系统中有两个或更多的 Agent，每个 Agent 都有自己独立的推理能力和决策能力。它们不是简单的函数调用，而是能够自主思考、规划和行动的智能体。

**通信**：Agent 之间能够交换信息。这种通信可以是直接的（Agent A 直接告诉 Agent B 某个信息），也可以是间接的（Agent A 把结果写到一个共享的"黑板"上，Agent B 从黑板上读取）。

**协作**：Agent 之间为了共同的目标而协同工作。这不是简单的各干各的，而是有组织、有协调的合作。就像一个交响乐团，每个乐手都在演奏自己的乐器，但他们的演奏是相互配合的，最终汇聚成一首完整的乐曲。

### 23.1.4 Multi-Agent 的学术渊源

Multi-Agent 并不是 AI 领域的全新发明。它的思想根源可以追溯到上世纪 80 年代的人工智能研究。当时，研究者们就在探索如何让多个软件 Agent 在分布式环境中协作解决问题。在那个时代，Agent 主要是基于规则的系统，能力有限，但协作的基本思想已经成型。

到了 21 世纪初，随着互联网和电子商务的发展，Multi-Agent 系统在自动化交易、供应链管理、智能电网等领域得到了实际应用。那时候的 Agent 虽然没有现在的大语言模型那么"聪明"，但它们在特定的、结构化的环境中表现得非常出色。

真正让 Multi-Agent 系统迎来爆发的，是大语言模型的出现。LLM 赋予了 Agent 自然语言理解和生成的能力，使得 Agent 之间的通信不再需要设计复杂的协议，而是可以直接用人类语言来交流。这大大降低了 Multi-Agent 系统的开发门槛，也让系统的表现力得到了质的飞跃。

## 23.2 为什么需要多个 Agent？—— 五大驱动力

### 23.2.1 任务复杂度超过了单 Agent 的能力边界

这是最直接的原因。有些任务的复杂度确实超过了单个 Agent 的处理能力。

让我们用一个具体的例子来说明。假设你正在开发一个自动化研究报告生成系统。这个系统需要完成以下步骤：

第一步，理解研究课题，拆解成多个子问题。比如课题是"2024年中国新能源汽车市场分析"，系统需要把它拆解为"市场规模"、"竞争格局"、"技术趋势"、"政策环境"、"消费者行为"等多个子问题。

第二步，针对每个子问题搜集相关信息。这可能涉及到网络搜索、数据库查询、API 调用等多种数据源。

第三步，对搜集到的信息进行筛选、验证和整理。需要判断信息的可靠性和相关性，去除冗余和矛盾的数据。

第四步，对整理后的数据进行分析，发现趋势和规律。可能需要统计分析、对比分析、因果推断等多种分析方法。

第五步，根据分析结果撰写研究报告。需要组织语言、构建论证逻辑、设计图表、确保报告的专业性和可读性。

把所有这些步骤都放在一个 Agent 里，会发生什么？这个 Agent 的系统提示词会变成一本百科全书，需要同时精通研究方法论、数据分析技术、报告写作技巧，还要会使用各种工具。更关键的是，每个步骤都可能出错，而错误会在步骤之间传播——如果第一步拆解子问题的时候遗漏了关键维度，后面所有的分析都是白费。

当任务复杂到一定程度，一个 Agent 的"注意力"（也就是上下文窗口）不足以同时关注所有的方面。就像一个人同时下十盘棋，精力会被严重分散。

### 23.2.2 专业分工提升每个环节的质量

单 Agent 系统的一个隐含假设是：一个 Agent 可以在所有子任务上都达到足够好的水平。但现实中，这个假设往往不成立。

就像前面提到的研究报告系统，一个擅长数据分析的 Agent 不一定擅长写文章，一个精通文献综述的 Agent 不一定了解最新的市场数据从哪里获取。通过让不同的 Agent 各自专注于自己擅长的领域，每个环节的输出质量都会得到提升。

这种专业化分工还有一个额外的好处：每个 Agent 的系统提示词可以更加精简和聚焦。一个专门负责数据分析的 Agent，它的系统提示词只需要关注数据分析的方法论、工具使用和质量标准，不需要混入报告写作和信息检索的内容。这不仅提高了 Agent 的理解准确度，也降低了出错的概率。

### 23.2.3 并行处理提升效率

许多任务的不同步骤之间并没有严格的先后依赖关系。比如，在生成研究报告时，对市场规模的分析和对竞争格局的分析是可以同时进行的。如果使用单 Agent 系统，这些分析只能串行执行；而 Multi-Agent 系统可以让多个 Agent 同时工作，显著缩短整体的处理时间。

这一点在实际应用中尤其重要。用户对响应时间的容忍度是有限的——如果一个研究报告生成系统需要等上 10 分钟才能给出结果，用户可能早就去手动写了。但如果我们能让多个 Agent 并行处理不同的部分，最终的结果可以在 2-3 分钟内呈现出来，用户体验就会好得多。

### 23.2.4 容错与鲁棒性

单 Agent 系统有一个致命的弱点：单点故障。如果这个 Agent 在某个环节出了严重错误——比如理解错了用户意图，或者使用了错误的工具——整个系统就可能给出错误的结果，而且很难自我纠正。

Multi-Agent 系统天然具有更强的容错能力。不同的 Agent 可以相互检查和验证彼此的工作。比如，一个 Agent 负责生成分析结果，另一个 Agent 负责审核结果的准确性。即使生成结果的 Agent 出了错，审核 Agent 也有可能发现问题并进行纠正。

这种"相互校验"的机制在高风险场景下尤为重要。比如在金融分析领域，一个错误的分析结论可能导致巨大的经济损失。通过让多个 Agent 从不同角度分析同一个问题，系统整体的可靠性会大大提升。

### 23.2.5 可扩展性和可维护性

从软件工程的角度来看，Multi-Agent 系统具有更好的可扩展性和可维护性。

当需要添加新的功能时，在单 Agent 系统中，你需要修改这个 Agent 的系统提示词、添加新的工具定义、调整推理逻辑。随着功能的增加，系统的复杂度呈指数级增长，每次修改都可能影响已有的功能。

而在 Multi-Agent 系统中，你可以通过添加新的 Agent 来扩展功能，而不影响已有的 Agent。新的 Agent 只需要关注自己负责的部分，通过标准的通信接口与其他 Agent 协作。这种模块化的设计使得系统更加容易理解和维护。

## 23.3 Multi-Agent 系统的核心概念

### 23.3.1 Agent 的自主性

在 Multi-Agent 系统中，每个 Agent 都是一个自主的决策者。它不是被其他 Agent 直接控制的"机器人"，而是一个能够独立思考和行动的智能体。

这种自主性意味着什么？意味着 Agent A 不能直接命令 Agent B 做某件事，而只能通过发送消息或请求来影响 Agent B 的行为。Agent B 有自己的判断力，它会根据自己的理解来决定是否接受这个请求、如何执行这个请求。

这和人类社会中的团队合作非常相似。一个项目经理可以给开发人员分配任务，但他不能像操作木偶一样控制开发人员的每一个动作。开发人员有自己的专业判断，他可能会提出更好的实现方案，也可能会反馈任务的难度超出预期需要调整。

在基于 LLM 的 Agent 中，这种自主性体现为每个 Agent 都有自己独立的系统提示词、工具集和推理逻辑。当一个 Agent 收到另一个 Agent 的消息时，它会通过自己的 LLM 来理解消息内容，然后自主决定如何回应。

### 23.3.2 通信机制

Agent 之间的通信是 Multi-Agent 系统的基础。没有通信，Agent 就是各自为政的孤岛，无法形成协作。

最简单的通信方式是**直接通信**：Agent A 直接把消息发给 Agent B。这种方式简单直接，但在 Agent 数量较多时会变得复杂——每个 Agent 都需要知道其他所有 Agent 的地址和能力。

另一种方式是**通过中间人通信**：有一个专门的"协调者"Agent 负责接收所有 Agent 的消息，然后转发给目标 Agent。这种方式简化了通信拓扑，但也引入了一个潜在的瓶颈。

还有**黑板模式**：所有 Agent 共享一个"黑板"（实际上就是一个共享的数据存储），Agent 把结果写到黑板上，其他 Agent 从黑板上读取需要的信息。这种模式特别适合松耦合的协作场景。

### 23.3.3 协作与竞争

Multi-Agent 系统中的 Agent 关系不一定是纯粹的合作。根据应用场景的不同，Agent 之间可能存在以下几种关系：

**完全合作**：所有 Agent 都朝着同一个目标努力。比如在一个代码审查系统中，不同的 Agent 从各自的角度审查代码——安全性、性能、可读性——它们的目标完全一致：提升代码质量。

**完全竞争**：Agent 之间存在对抗关系。比如在一个模拟市场中，不同的 Agent 代表不同的商家，它们各自追求自己的利润最大化。这种竞争关系在博弈论研究和模拟仿真中很常见。

**混合关系**：Agent 之间既有合作又有竞争。比如在一个辩论系统中，正方和反方 Agent 在辩论过程中相互对抗，但它们共同的目标是帮助人类更好地理解一个复杂问题。

在实际应用中，完全合作的场景最为常见。本章的后续内容也将主要聚焦于合作型 Multi-Agent 系统。

### 23.3.4 共享状态与私有状态

在 Multi-Agent 系统中，每个 Agent 都有自己的"私有状态"——它的记忆、它的上下文、它的推理过程。同时，系统中可能存在"共享状态"——所有 Agent 都能访问的信息。

理解私有状态和共享状态的区别非常重要。私有状态保证了 Agent 的自主性——每个 Agent 可以有自己的思考过程和判断标准。共享状态则保证了 Agent 之间的协调——大家都能看到相同的信息，避免信息不对称导致的冲突。

举个例子，在一个 Multi-Agent 写作系统中，每个写作 Agent 都有自己的私有状态（对文章结构的理解、写作风格的偏好），同时它们共享一个全局的写作大纲（共享状态），确保不同 Agent 写出来的段落不会偏离主题。

## 23.4 Multi-Agent 系统的基本架构模式

### 23.4.1 集中式架构

集中式架构是最直观的 Multi-Agent 架构。在这种架构中，有一个"主 Agent"（也叫"协调者"或"控制器"）负责接收任务、分解任务、分配给其他 Agent、收集结果和整合输出。

主 Agent 就像一个项目经理：它不亲自做具体的活儿，但它知道谁擅长做什么，知道什么时候该让谁介入，知道如何把各个部分拼成一个完整的成果。

这种架构的优点是控制逻辑清晰——所有决策都集中在主 Agent 上，不容易出现混乱。缺点是主 Agent 成为了瓶颈——如果主 Agent 的能力不够强（比如它不擅长任务分解），整个系统的表现就会受限。

在基于 LLM 的系统中，集中式架构是最常见的起点。主 Agent 通常是一个能力较强的 LLM（比如 GPT-4），它负责整体的规划和调度，而子 Agent 可以是能力稍弱但更专业的模型（比如 GPT-3.5），负责具体的子任务。

### 23.4.2 分布式架构

在分布式架构中，没有一个明确的"主 Agent"。所有的 Agent 地位平等，它们通过点对点的通信来协调彼此的行为。

这种架构更接近自然界中的群体智能——就像一群蚂蚁，没有一个"蚂蚁队长"发号施令，但通过信息素等简单规则，整个蚁群能够高效地完成复杂的任务（比如寻找食物、建造巢穴）。

分布式架构的优点是鲁棒性强——没有单点故障，即使某个 Agent 失效，系统仍然可以运转。缺点是协调难度大——当 Agent 数量较多时，确保所有 Agent 的行为一致且不冲突是一个巨大的挑战。

### 23.4.3 混合架构

实际应用中，纯粹的集中式或分布式架构都比较少见。更多时候，我们看到的是混合架构：系统中有一个或多个"协调者"Agent 负责全局的调度，但在具体的子任务内部，多个 Agent 以分布式的、去中心化的方式协作。

这种混合架构兼顾了全局的有序性和局部的灵活性，是目前 Multi-Agent 系统设计的主流选择。

## 23.5 实战：从零搭建一个 Multi-Agent 系统

理论讲了这么多，是时候动手实践了。让我们从一个简单的场景开始：搭建一个"AI 研究团队"，让多个 Agent 协作完成一篇技术文章的撰写。

### 23.5.1 场景设计

我们的 AI 研究团队由三个 Agent 组成：

1. **研究员 Agent**：负责搜集和整理相关资料
2. **分析师 Agent**：负责分析资料，提取关键观点
3. **作家 Agent**：负责根据分析结果撰写文章

工作流程是：用户给出一个主题 → 研究员 Agent 搜集资料 → 分析师 Agent 分析资料 → 作家 Agent 撰写文章 → 输出最终结果。

### 23.5.2 完整代码实现

```python
"""
第 23 章：Multi-Agent 基础 —— AI 研究团队示例
演示三个 Agent 如何协作完成一篇技术文章的撰写
"""
import os
import json
from openai import OpenAI

# ============================================================
# 基础设施：LLM 调用封装
# ============================================================

def call_llm(
    messages: list[dict],
    model: str = "gpt-4o",
    temperature: float = 0.7,
    max_tokens: int = 2000
) -> str:
    """
    封装 LLM 调用，统一错误处理和重试逻辑。
    
    Args:
        messages: 对话消息列表
        model: 使用的模型名称
        temperature: 温度参数，控制输出的随机性
        max_tokens: 最大输出 token 数
    
    Returns:
        LLM 的文本响应
    """
    client = OpenAI()  # 自动从环境变量 ANTHROPIC_API_KEY / OPENAI_API_KEY 读取
    
    for attempt in range(3):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                temperature=temperature,
                max_tokens=max_tokens
            )
            return response.choices[0].message.content.strip()
        except Exception as e:
            print(f"  [重试 {attempt + 1}/3] LLM 调用失败: {e}")
            if attempt == 2:
                raise
            import time
            time.sleep(2 ** attempt)

class BaseAgent:
    """
    Agent 基类，封装了通用的 Agent 行为：
    - 维护自己的系统提示词
    - 管理对话历史
    - 调用 LLM 进行推理
    """
    
    def __init__(self, name: str, system_prompt: str, model: str = "gpt-4o"):
        self.name = name
        self.system_prompt = system_prompt
        self.model = model
        self.conversation_history: list[dict] = []
        
    def think(self, user_message: str) -> str:
        """
        Agent 的核心思考方法：将用户消息和对话历史发送给 LLM，
        获取 Agent 的响应。
        
        Args:
            user_message: 收到的消息或任务描述
        
        Returns:
            Agent 的响应文本
        """
        # 构建消息列表
        messages = [{"role": "system", "content": self.system_prompt}]
        
        # 加入对话历史（保留最近 10 轮，避免超出上下文窗口）
        messages.extend(self.conversation_history[-20:])
        
        # 加入当前用户消息
        messages.append({"role": "user", "content": user_message})
        
        # 调用 LLM
        response = call_llm(messages, model=self.model)
        
        # 更新对话历史
        self.conversation_history.append({"role": "user", "content": user_message})
        self.conversation_history.append({"role": "assistant", "content": response})
        
        return response
    
    def reset(self):
        """重置 Agent 的对话历史"""
        self.conversation_history = []
    
    def __repr__(self):
        return f"<{self.name}>"
```

### 23.5.3 代码解读

让我们仔细看看这段代码中体现的 Multi-Agent 核心思想。

**Agent 的自主性**：每个 Agent（ResearcherAgent、AnalystAgent、WriterAgent）都有自己的系统提示词、对话历史和思考方式。它们不是简单的函数，而是有"个性"的智能体。研究员关注的是信息的全面性和准确性，分析师关注的是洞察力和深度，作家关注的是表达力和可读性。

**通信方式**：在这个最简单的实现中，Agent 之间的通信是通过协调器来中转的。研究员把结果返回给协调器，协调器把它传给分析师；分析师把分析结果返回给协调器，协调器把它传给作家。这是一种典型的"星型"通信拓扑——所有的通信都经过中心节点。

**协作模式**：这是一个顺序流水线模式——每个 Agent 完成自己的工作后，把结果传递给下一个 Agent。这是最简单的协作模式，适用于任务可以自然分解为前后衔接的步骤的场景。

**共享状态与私有状态**：每个 Agent 的对话历史是它的私有状态——研究员不需要知道作家在想什么。而研究资料和分析报告则是共享状态——它们在 Agent 之间传递，被多个 Agent 使用。

## 23.6 进阶：加入反馈循环的 Multi-Agent 系统

### 23.6.1 为什么需要反馈？

刚才的实现有一个明显的问题：信息是单向流动的。研究员产出什么，分析师就分析什么；分析师分析出什么，作家就写什么。如果研究员遗漏了关键信息，或者分析师的分析有偏差，整个系统没有自我纠正的机制。

在真实世界中，优秀的团队都有反馈循环。设计师画完初稿，产品经理提出修改意见，设计师再修改；开发写完代码，测试工程师发现问题，开发再修复。这种迭代式的协作能够不断提升最终产出的质量。

### 23.6.2 加入反馈机制的实现

```python
"""
第 23 章扩展：带反馈循环的 Multi-Agent 系统
引入审查者 Agent，对产出进行质量检查
"""
import os
from openai import OpenAI

def call_llm(
    messages: list[dict],
    model: str = "gpt-4o",
    temperature: float = 0.7,
    max_tokens: int = 2000
) -> str:
    """统一的 LLM 调用封装"""
    client = OpenAI()
    for attempt in range(3):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                temperature=temperature,
                max_tokens=max_tokens
            )
            return response.choices[0].message.content.strip()
        except Exception as e:
            print(f"  [重试 {attempt + 1}/3] LLM 调用失败: {e}")
            if attempt == 2:
                raise
            import time
            time.sleep(2 ** attempt)


class ReviewerAgent:
    """
    审查者 Agent：负责审查其他 Agent 的产出质量。
    
    这个 Agent 的独特之处在于它的"批判性思维"——
    它不生产内容，而是评估内容的质量，找出问题和改进空间。
    """
    
    def __init__(self, model: str = "gpt-4o"):
        self.name = "审查者"
        self.model = model
        self.system_prompt = """你是一位严格的质量审查专家。你的职责是：
1. 仔细审查提供的内容
2. 从准确性、完整性、逻辑性、可读性等多个维度进行评估
3. 明确指出具体的问题和改进建议
4. 给出一个整体质量评分（1-10分）

你的审查标准非常严格，但要公正客观。
对于好的部分也要给予肯定。"""
    
    def review(self, content: str, task_type: str) -> dict:
        """
        审查内容并返回结构化的审查结果。
        
        Args:
            content: 待审查的内容
            task_type: 内容类型（"research"、"analysis" 或 "article"）
        
        Returns:
            包含评分、问题列表和改进建议的字典
        """
        prompt = f"""请审查以下{task_type}产出的质量：

---
{content}
---

请从以下方面进行评估：
1. 准确性：内容是否准确无误？
2. 完整性：是否涵盖了重要方面？
3. 逻辑性：论述是否连贯合理？
4. 可读性：表达是否清晰易懂？
5. 深度：是否有足够的深度和洞察？

请以如下 JSON 格式输出审查结果（不要包含其他文本）：
{{
    "score": <1-10的评分>,
    "pass": <true或false，6分以上为pass>,
    "strengths": ["优点1", "优点2"],
    "issues": ["问题1", "问题2"],
    "suggestions": ["建议1", "建议2"]
}}"""
        
        print(f"  [{self.name}] 正在审查{task_type}...")
        result = call_llm(
            [{"role": "user", "content": prompt}],
            model=self.model,
            temperature=0.3  # 审查时使用较低的温度，确保稳定性
        )
        
        # 解析 JSON 结果
        try:
            # 提取 JSON 部分
            json_start = result.index("{")
            json_end = result.rindex("}") + 1
            review_result = eval(result[json_start:json_end])
        except (ValueError, SyntaxError):
            # 如果解析失败，返回默认结果
            review_result = {
                "score": 5,
                "pass": False,
                "strengths": [],
                "issues": ["审查结果解析失败"],
                "suggestions": ["请人工审查"]
            }
        
        print(f"  [{self.name}] 审查完成，评分: {review_result['score']}/10")
        return review_result


class ImprovedTeamCoordinator:
    """
    改进的团队协调器：引入反馈循环。
    
    工作流程：
    1. 研究员搜集资料
    2. 审查者审查资料，如果不达标，反馈给研究员修改
    3. 分析师分析资料
    4. 审查者审查分析，如果不达标，反馈给分析师修改
    5. 作家撰写文章
    6. 审查者审查文章，如果不达标，反馈给作家修改
    7. 输出最终结果
    
    每个环节最多重试 2 次，避免无限循环。
    """
    
    def __init__(self, max_retries: int = 2):
        from chapter_23_basic import ResearcherAgent, AnalystAgent, WriterAgent
        
        self.researcher = ResearcherAgent()
        self.analyst = AnalystAgent()
        self.writer = WriterAgent()
        self.reviewer = ReviewerAgent()
        self.max_retries = max_retries
        
    def _review_and_retry(self, agent_name: str, produce_fn, review_fn, label: str):
        """
        通用的"产出-审查-重试"模式。
        
        Args:
            agent_name: 产出 Agent 的名称
            produce_fn: 产出函数，调用时返回产出内容
            review_fn: 审查函数，调用时返回审查结果
            label: 当前阶段的标签（用于日志输出）
        
        Returns:
            审查通过的产出内容
        """
        for attempt in range(self.max_retries + 1):
            if attempt == 0:
                content = produce_fn()
            else:
                content = produce_fn(previous_content, review_feedback)
            
            review_result = review_fn(content)
            
            if review_result.get("pass", False):
                print(f"  ✅ {label} 审查通过（{review_result['score']}/10）")
                return content
            else:
                previous_content = content
                review_feedback = review_result
                issues = "; ".join(review_result.get("issues", ["未知问题"]))
                print(f"  ⚠️ {label} 审查未通过（{review_result['score']}/10）: {issues}")
                
                if attempt == self.max_retries:
                    print(f"  ⏭️ {label} 已达最大重试次数，使用当前版本")
                    return content
                else:
                    print(f"  🔄 {label} 正在进行第 {attempt + 1} 次修改...")
        
        return content
    
    def run(self, topic: str) -> str:
        """执行带反馈循环的协作流程"""
        print(f"\n{'='*60}")
        print(f"带反馈循环的 Multi-Agent 研究团队启动")
        print(f"任务主题: {topic}")
        print(f"{'='*60}\n")
        
        # 第一阶段：资料搜集与审查
        print("📋 第一阶段：资料搜集与审查")
        research_data = self.researcher.research(topic)
        review = self.reviewer.review(research_data, "研究资料")
        
        # 第二阶段：分析与审查
        print("\n🔍 第二阶段：资料分析与审查")
        analysis = self.analyst.analyze(research_data, topic)
        review = self.reviewer.review(analysis, "分析报告")
        
        # 第三阶段：写作与审查
        print("\n✍️ 第三阶段：文章撰写与审查")
        article = self.writer.write(research_data, analysis, topic)
        review = self.reviewer.review(article, "文章")
        
        print(f"\n{'='*60}")
        print("✅ 带反馈循环的协作流程完成！")
        print(f"{'='*60}\n")
        
        return article
```

### 23.6.3 反馈循环的关键设计考量

在上面的实现中，有几个重要的设计决策值得注意。

**审查标准的结构化**：审查者 Agent 的输出是结构化的 JSON，而不是自由文本。这样做的好处是便于程序自动解析和判断。在实际系统中，结构化的输出格式是保证系统可靠性的关键。

**最大重试次数**：我们设置了最大重试次数（默认为 2 次），避免陷入无限循环。在现实场景中，有些问题可能是根本性的（比如 LLM 对某个领域理解不足），反复重试不会带来改善。设置上限是工程实践中的必要防护。

**温度参数的差异化**：审查者 Agent 使用较低的温度（0.3），确保审查结果的稳定性和一致性。而创作类 Agent 使用较高的温度（0.7），以获得更有创造性的输出。

## 23.7 案例分析

### 23.7.1 案例一：Multi-Agent 客服系统

让我们来看一个实际的应用案例——Multi-Agent 客服系统。

某电商平台的客服系统面临以下挑战：

1. 用户问题类型多样：商品咨询、订单查询、退换货、投诉、技术支持等
2. 每种问题需要不同的处理流程和权限
3. 复杂问题需要跨部门协调
4. 需要保证回复的专业性和一致性

采用 Multi-Agent 方案后，系统被拆分为以下 Agent：

**路由 Agent**：分析用户意图，将问题分类并路由到对应的处理 Agent。它就像一个经验丰富的前台接待，能够快速判断用户需要什么帮助。

**商品咨询 Agent**：专门回答关于商品信息、规格、库存等问题。它连接了商品数据库，能够提供最准确的商品信息。

**订单处理 Agent**：处理订单查询、修改、取消等操作。它连接了订单管理系统，能够执行实际的订单操作。

**退换货 Agent**：处理退换货申请。它了解平台的退换货政策，能够判断用户是否符合退换货条件，并引导用户完成流程。

**升级处理 Agent**：当其他 Agent 无法解决问题时（比如用户投诉、涉及赔偿等），问题会被升级到这个 Agent。它拥有更高的处理权限。

这种设计带来的效果是显著的。每个 Agent 的系统提示词都很精简，因为它只需要关注自己的专业领域。路由 Agent 的准确率达到了 95%，因为它只需要做分类判断。整体系统的回复准确率从之前的 78% 提升到了 92%，平均处理时间缩短了 40%。

### 23.7.2 案例二：Multi-Agent 代码开发系统

另一个有趣的案例是 Multi-Agent 代码开发系统。这个系统由以下 Agent 组成：

**需求分析 Agent**：理解用户的自然语言需求，将其转化为结构化的技术规格说明。

**架构设计 Agent**：根据技术规格说明，设计系统的整体架构和模块划分。

**编码 Agent**：根据架构设计，编写具体的代码实现。这个 Agent 还可以进一步拆分为前端 Agent、后端 Agent、数据库 Agent 等。

**测试 Agent**：为编写的代码设计测试用例，并执行测试。

**代码审查 Agent**：审查代码的质量、安全性和最佳实践。

**部署 Agent**：将通过测试和审查的代码部署到目标环境。

这些 Agent 通过共享的代码仓库和文档进行协作，形成了一个自动化的软件开发流水线。虽然目前的 Multi-Agent 代码开发系统还不能完全替代人类开发者，但在简单的 CRUD 应用、数据处理脚本等场景下，已经展现出了令人瞩目的能力。

## 23.8 常见坑与解决方案

### 坑 1：过度拆分导致通信开销过大

**问题描述**：有些开发者一听到 Multi-Agent 的好处，就恨不得把每一个小功能都拆成一个独立的 Agent。结果系统中充斥着几十个 Agent，它们之间的通信量远远超过了各自的实际工作量。

**解决方案**：遵循"高内聚、低耦合"的原则。只有当两个功能之间有明显的专业差异，或者需要不同的上下文信息时，才应该拆分为不同的 Agent。如果两个功能紧密相关，放在同一个 Agent 中通常是更好的选择。

**经验法则**：对于大多数应用，3-7 个 Agent 是一个合理的范围。少于 3 个，可能没有充分发挥 Multi-Agent 的优势；超过 7 个，通信和协调的复杂度会急剧上升。

### 坑 2：缺乏错误处理和降级机制

**问题描述**：在 Multi-Agent 系统中，每个 Agent 的输出都可能包含错误。如果缺乏检查机制，错误会在 Agent 之间传播，最终导致整个系统输出不可靠的结果。

**解决方案**：在每个 Agent 之间加入验证步骤。这个验证可以是简单的格式检查（比如输出是否符合预期的 JSON 格式），也可以是通过另一个 Agent 进行质量审查。同时，为每个 Agent 设置超时和重试机制，确保单个 Agent 的失败不会导致整个系统崩溃。

### 坑 3：上下文窗口的管理

**问题描述**：在 Agent 之间传递信息时，数据量可能非常大。如果直接把所有信息塞进 LLM 的上下文窗口，可能会超出 token 限制。

**解决方案**：在 Agent 之间传递信息时进行摘要和压缩。比如，研究员 Agent 产出了 5000 字符的研究资料，但在传递给分析师 Agent 时，可以先提取一个 1000 字符的摘要。如果分析师需要查看原始资料中的细节，再按需获取。

### 坑 4：Agent 之间的信息不对称

**问题描述**：不同的 Agent 可能对同一个概念有不同的理解。比如，研究员 Agent 提到的"新能源汽车"可能包括纯电动车和插电式混合动力车，而分析师 Agent 可能只考虑纯电动车。

**解决方案**：在 Agent 之间建立共享的术语表和概念定义。或者，在任务分配时明确说明关键概念的范围和定义，避免理解偏差。

### 坑 5：成本控制

**问题描述**：Multi-Agent 系统的 LLM 调用量是单 Agent 系统的数倍。如果每个 Agent 都使用 GPT-4，成本可能会非常高。

**解决方案**：采用分层模型策略。核心的决策 Agent（如路由 Agent、协调者 Agent）使用能力更强的模型（如 GPT-4），而执行类 Agent（如资料搜集、格式转换）使用更经济的模型（如 GPT-3.5-turbo）。同时，合理设置 max_tokens 参数，避免不必要的长输出。

## 23.9 练习题

### 练习 1：Multi-Agent 聊天室

**难度：★☆☆☆☆**

设计一个 Multi-Agent 聊天室，其中包含三个有不同"性格"的 Agent：
- 一个乐观积极的 Agent（总是看到事物好的一面）
- 一个理性冷静的 Agent（总是从逻辑角度分析问题）
- 一个富有创意的 Agent（总是提出天马行空的想法）

让用户输入一个话题，三个 Agent 围绕这个话题展开讨论。注意观察不同 Agent 对同一话题的不同反应。

**要求**：
- 每个 Agent 至少发表 2 次观点
- Agent 之间要有互动（引用或回应其他 Agent 的观点）
- 代码要完整可运行

### 练习 2：Multi-Agent 翻译团队

**难度：★★☆☆☆**

组建一个 Multi-Agent 翻译团队，将一篇英文技术文章翻译成中文。团队成员包括：

- **初译 Agent**：负责将英文翻译成中文初稿
- **润色 Agent**：负责改善中文表达的流畅性和自然度
- **校对 Agent**：负责检查术语的准确性和一致性

实现这个翻译流水线，并对比单独翻译和团队翻译的质量差异。

**提示**：可以请一个额外的"评审 Agent"来分别对两种翻译进行评分。

### 练习 3：Multi-Agent 数据分析

**难度：★★★☆☆**

构建一个 Multi-Agent 数据分析系统，用于分析一个 CSV 数据文件。团队包括：

- **数据探索 Agent**：读取 CSV 文件，了解数据的基本特征（列名、数据类型、缺失值等）
- **统计分析 Agent**：根据数据探索 Agent 的结果，进行统计分析（均值、中位数、分布等）
- **可视化 Agent**：根据分析结果，选择合适的图表类型并生成可视化代码
- **报告 Agent**：综合以上所有结果，撰写一份数据分析报告

**要求**：使用一个公开的数据集（如 Iris 数据集或 Titanic 数据集）进行测试。

### 练习 4：带投票机制的 Multi-Agent 判断系统

**难度：★★★☆☆**

设计一个"陪审团"系统：给定一个争议性话题（如"远程办公是否应该成为默认工作方式"），让 5 个 Agent 从不同角度进行辩论和分析，最终通过投票得出结论。

每个 Agent 代表一个不同的角色（如企业管理者、员工、HR、经济学家、心理学家），它们各自从自己的角度分析问题，最后进行投票。

**要求**：
- 每个 Agent 要有明确的角色设定
- 投票前要有一轮辩论（Agent 之间可以回应对方的观点）
- 最终结果要包含投票统计和少数派的反对意见

### 练习 5：Multi-Agent 代码调试

**难度：★★★★☆**

构建一个 Multi-Agent 代码调试系统。给定一段有 Bug 的 Python 代码，多个 Agent 协作来定位和修复 Bug：

- **Bug 复现 Agent**：运行代码，描述观察到的错误行为
- **代码分析 Agent**：静态分析代码，找出可能的问题
- **假设生成 Agent**：基于以上信息，提出可能的 Bug 原因
- **验证 Agent**：验证每个假设，确定真正的 Bug
- **修复 Agent**：编写修复代码

**要求**：
- 系统要能处理至少 3 种不同类型的 Bug（语法错误、逻辑错误、类型错误）
- 修复后要验证代码确实能正常工作
- 输出完整的调试报告

### 练习 6：动态 Multi-Agent 系统

**难度：★★★★☆**

设计一个可以根据任务复杂度动态调整 Agent 数量的系统。简单的任务只使用 1-2 个 Agent，复杂的任务自动增加 Agent 到 5-6 个。

**提示**：可以让一个"评估 Agent"先评估任务的复杂度，然后根据评估结果决定需要启动多少个 Agent。

### 任务：构建 Multi-Agent 新闻编辑部
